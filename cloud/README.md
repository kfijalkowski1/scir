# SCIR Cloud Infrastructure

This directory contains the OpenTofu + Terragrunt implementation for the washing machine IoT monitor.

## Toolchain

- OpenTofu language compatibility: Terraform `1.14`
- AWS provider: `~> 6.0`
- Terragrunt: latest (v1.x)

## Layout

- `root.hcl` - global Terragrunt configuration (provider generation and S3 remote state)
- `environments/prod/*` - Terragrunt units and dependency graph
- `modules/*` - reusable Terraform modules

## IAM user setup

Create an IAM user with `AdministratorAccess` IAM policy attached, create security credentials and add them as a profile to your local AWS CLI. This setup will **not** work using the root IAM user!

Create the IAM user:

```shell
# assuming you have a working AWS cli with root access

# create the IAM user
aws iam create-user --user-name terraform

# create the force MFA IAM policy
aws iam create-policy --policy-name ForceMFA --policy-document file://misc/force-mfa-policy.json

# attach policies
aws iam attach-user-policy --user-name test --policy-arn "arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):policy/ForceMFA"
aws iam attach-user-policy --user-name test --policy-arn "arn:aws:iam::aws:policy/AdministratorAccess"

# NOTE DOWN THE ACCESS KEY ID AND SECRET ACCESS KEY!
aws iam create-access-key --user-name terraform
```

Now create an MFA device for the user:

```shell
aws iam create-virtual-mfa-device --virtual-mfa-device-name terraform-personal-mfa-device --bootstrap-method QRCodePNG --outfile activation.png
```

Scan the QR code in `activation.png` using your favorite MFA app.

Finally, activate the MFA device and associate it with the user. You will need to enter two consecutive MFA codes:

```shell
read -s CODE1
# type in code 1
read -s CODE2
# type in code 2
aws iam enable-mfa-device \
  --user-name terraform \
  --serial-number "arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):mfa/terraform-personal-mfa-device" \
  --authentication-code1 "$CODE1" \
  --authentication-code2 "$CODE2"
```

You can now add the user to you AWS CLI:

```shell
READ -s ACCESS_KEY_ID
# paste the access key_ID
READ -s SECRET_ACCESS_KEY
# paste the secret access key

echo "
[profile terraform_mfa]
aws_access_key_id = $ACCESS_KEY_ID
aws_secret_access_key = $SECRET_ACCESS_KEY
mfa_serial = arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):mfa/terraform-personal-mfa-device

[profile terraform]
region = eu-central-1
" | tee -a "$HOME/.aws/config"
```

Try to do some action using the new user. The second command should give you access denied:

```shell
AWS_PROFILE=terraform aws sts get-caller-identity
AWS_PROFILE=terraform aws s3 ls
```

Now enter a generated MFA token and retry, the command should now work:

```shell
# Note: different profile used for MFA!
AWS_PROFILE=terraform_mfa aws configure mfa-login
# when asked for a profile to update, type "terraform"

# this should now work
AWS_PROFILE=terraform aws s3 ls
```

## Before first apply

1. Export required environment variables:

```bash
export SCIR_DISCORD_WEBHOOK_URL="https://discord.com/api/webhooks/..."
export SCIR_WEBHOOK_AUTH_TOKEN="<long-random-token>"
```

2. Move into the environment folder:

```bash
cd environments/prod
```

3. Initialize and apply in dependency order using Terragrunt:

```bash
terragrunt run --all --backend-bootstrap --working-dir . plan
terragrunt run --all --backend-bootstrap --working-dir . apply
```

## Notes

- Time series data is stored as CloudWatch custom metrics.
  - Readings metric: `WasherPowerReading` in namespace `SCIR/Washer`
  - Events metric: `WasherEventCode` in namespace `SCIR/Washer`
  - Event code mapping: `1=cycle_start`, `2=cycle_end`, `3=buzzer_silence`
  - Processor Lambda writes metrics with `PutMetricData` and evaluates cycle state/window with `GetMetricData`.
- MQTT topics are generated as:
  - Telemetry filter: `scir/prod/washer/+/status/switch:0`
  - Control/events topic: `scir/prod/washer/buzzer/events`
- IoT certificates are generated by Terraform in this implementation.
  - Private keys will be stored in state; protect state access strictly.
- CloudWatch dashboard resource is managed by Terraform.
  - Public share link setup may still require a manual AWS Console action.

## MQTT test plan (Shelly + consumer board emulation)

Use standard MQTT CLI tools:

- `mosquitto_pub`
- `mosquitto_sub`

Install (Debian/Ubuntu):

```bash
sudo apt-get update && sudo apt-get install -y mosquitto-clients
```

### 1 Prepare connection variables

Use the IoT Data-ATS endpoint and certificates generated by Terraform:

```bash
export MQTT_HOST="<your-iot-data-ats-endpoint>"
export MQTT_PORT=8883
export MQTT_CA="/path/to/AmazonRootCA1.pem"
export MQTT_CERT="/path/to/device-certificate.pem.crt"
export MQTT_KEY="/path/to/private.pem.key"

export TELEMETRY_TOPIC="scir/prod/washer/shelly-plug/status/switch:0"
export CONTROL_TOPIC="scir/prod/washer/buzzer/events"
```

```shell
mkdir -p ~/certs
curl -o ~/certs/AmazonRootCA1.pem https://www.amazontrust.com/repository/AmazonRootCA1.pem
chmod 644 ~/certs/AmazonRootCA1.pem
ls -l ~/certs

export MQTT_CA="$HOME/certs/AmazonRootCA1.pem"
export MQTT_HOST=$(terragrunt output -raw iot_data_endpoint)
export MQTT_PORT=8883

terragrunt output -raw shelly_certificate_pem | tee shelly_cert.crt
export MQTT_SHELLY_CERT="$(pwd)/shelly_cert.crt"
terragrunt output -raw shelly_private_key | tee shelly_key.key
export MQTT_SHELLY_KEY="$(pwd)/shelly_key.key"

terragrunt output -raw esp_certificate_pem | tee esp_cert.crt
export MQTT_ESP_CERT="$(pwd)/esp_cert.crt"
terragrunt output -raw esp_private_key | tee esp_key.key
export MQTT_ESP_KEY="$(pwd)/esp_key.key"

export TELEMETRY_TOPIC="scir/prod/washer/shelly-plug/status/switch:0"
export CONTROL_TOPIC="scir/prod/washer/buzzer/events"

export MQTT_SHELLY_CLIENT_ID=scir-prod-shelly-plug
export MQTT_ESP_CLIENT_ID=scir-prod-esp32-buzzer
```

If your deployment uses different values, read them from Terragrunt/Terraform outputs:

```bash
cd environments/prod/iot
terragrunt output -raw shelly_publish_topic
terragrunt output -raw control_topic
```

### 2 Emulate the consumer board (ESP32)

Listen on the control/events topic:

```bash
mosquitto_sub -h "$MQTT_HOST" -p "$MQTT_PORT" \
  --cafile "$MQTT_CA" --cert "$MQTT_ESP_CERT" --key "$MQTT_ESP_KEY" \
  -q 1 -t "$CONTROL_TOPIC" -v -i "$MQTT_ESP_CLIENT_ID"
```

This should receive events like `cycle_start`, `cycle_end`, and `buzzer_silence`.

### 3 Emulate Shelly smart plug telemetry

Publish power readings to telemetry topic:

```bash
# format accepted by this project: top-level apower
mosquitto_pub -h "$MQTT_HOST" -p "$MQTT_PORT" \
  --cafile "$MQTT_CA" --cert "$MQTT_SHELLY_CERT" --key "$MQTT_SHELLY_KEY" \
  -q 1 -t "$TELEMETRY_TOPIC" \
  -m '{"apower": 1250.4}' -i "$MQTT_SHELLY_CLIENT_ID"

# also accepted by this project: nested switch:0.apower
mosquitto_pub -h "$MQTT_HOST" -p "$MQTT_PORT" \
  --cafile "$MQTT_CA" --cert "$MQTT_SHELLY_CERT" --key "$MQTT_SHELLY_KEY" \
  -q 1 -t "$TELEMETRY_TOPIC" \
  -m '{"switch:0":{"apower": 1250.4}}' -i "$MQTT_SHELLY_CLIENT_ID"
```

To trigger cycle detection with defaults:

- publish high power (>= `10W`) to trigger `cycle_start`
- then keep power low (<= `3W`) for ~`180s` to allow `cycle_end`

These are current processor defaults (`START_POWER_THRESHOLD`, `END_POWER_THRESHOLD`, `LOW_POWER_WINDOW_SECONDS`) and may be tuned in future deployments.
They are defined in `cloud/modules/lambda-processor/src/handler.py` and set through Lambda environment variables from `cloud/modules/lambda-processor/main.tf`.

### 4 Emulate user silencing the buzzer (board button / app webhook effect)

Publish silence event directly to control topic:

```bash
mosquitto_pub -h "$MQTT_HOST" -p "$MQTT_PORT" \
  --cafile "$MQTT_CA" --cert "$MQTT_CERT" --key "$MQTT_KEY" \
  -q 1 -t "$CONTROL_TOPIC" \
  -m '{"event_type":"buzzer_silence","action":"buzzer_off","source":"esp32","device_id":"washing-machine","ts":1735689600000}'
```

## Expected MQTT message schema

### Telemetry from Shelly (Gen3-compatible topic)

Topic pattern used by this stack:

```text
scir/prod/washer/<device>/status/switch:0
```

Payload accepted by the processor:

```json
{
  "apower": 123.4,
  "ingest_ts": 1735689600000
}
```

or

```json
{
  "switch:0": {
    "apower": 123.4
  }
}
```

Notes:

- `apower` is required (top-level or under `switch:0`) for ingestion
- `ingest_ts` is optional (if omitted, processing time is used)

### Control/events topic payload

Topic:

```text
scir/prod/washer/buzzer/events
```

Payload shape:

```json
{
  "event_type": "cycle_start | cycle_end | buzzer_silence",
  "action": "cycle_started | buzzer_on | buzzer_off",
  "source": "lambda_processor | api_webhook | esp32",
  "device_id": "washing-machine",
  "ts": 1735689600000
}
```
